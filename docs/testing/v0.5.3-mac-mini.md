## Test 2 — LAN access: "Invalid host header" fix

**What we're testing:** Accessing the WebUI from a remote machine by IP was blocked by TrustedHostMiddleware. Fix: auto-enumerate local interface IPs when `host: 0.0.0.0` and add them to the allowlist at startup.

Prior behavior: `http://10.0.100.203:8764` returned `Invalid host header`.

**Root cause:** TrustedHostMiddleware only allowed `localhost`, `127.0.0.1`, and `*.local`. Bare IP addresses in the Host header were rejected. Raw IPs are not a DNS rebinding vector (rebinding requires a domain name), so they can be safely allowed.

**Fix:** At startup, when `host == "0.0.0.0"`, enumerate local interface IPs via `psutil.net_if_addrs()` and append non-loopback IPv4 addresses to `allowed_hosts`.

Workaround tested before fix landed: adding `extra_allowed_hosts: [10.0.100.203]` to config.yaml.


Note: commit-pinned URL used for install.sh while GitHub raw CDN cached the pre-fix version (~10 min propagation delay).

### Bug found: BSD sed incompatibility (macOS)

`install.sh` used `sed -i "s/PLACEHOLDER/$KEY/" $CONFIG` which is GNU sed syntax. macOS ships BSD sed, which requires `sed -i '' ...`. This caused the install to fail at step 4 with:

```
sed: 1: "/Users/dylan/.llm-valet ...": extra characters at the end of d command
```

**Fix:** Switched from single-quoted heredoc + sed substitution to an unquoted heredoc with `$API_KEY` expanded directly — no sed needed. Commit `fb9417c`.
