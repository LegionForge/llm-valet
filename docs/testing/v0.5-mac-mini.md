## Test 3 — Fix: num_ctx preserved across PAUSE → RESUME

**What we're testing:** When a model is loaded with a non-default context window and then paused, resuming restores the same context length — not Ollama's built-in default.

**How it works:** `pause()` captures `context_length` from `/api/ps` before the `keep_alive: 0` eviction call. `resume()` passes `options: {num_ctx: N}` if a value was captured. Verifiable via `/api/ps context_length` after resume.

```bash
# Step 1: Load model at a specific non-default ctx (e.g. 16K = 16384)
curl -s -X POST http://localhost:8764/load \
  -H "Content-Type: application/json" \
  -H "X-API-Key: <api-key>" \
  -d '{"model": "qwen3.5:0.8b", "num_ctx": 16384}' | python3 -m json.tool

# Step 2: Confirm ctx loaded
curl -s http://localhost:11434/api/ps | python3 -m json.tool | grep context_length

# Step 3: PAUSE
curl -s -X POST http://localhost:8764/pause \
  -H "X-API-Key: <api-key>" | python3 -m json.tool

# Step 4: Confirm model evicted
curl -s http://localhost:11434/api/ps | python3 -m json.tool

# Step 5: RESUME
curl -s -X POST http://localhost:8764/resume \
  -H "X-API-Key: <api-key>" | python3 -m json.tool

# Step 6: Confirm ctx restored
curl -s http://localhost:11434/api/ps | python3 -m json.tool | grep context_length
```

## Test 4 — Feat: over-commit warning banner (model > RAM pause threshold)

**What we're testing:** When a loaded model's `memory_used_mb` exceeds `total_ram × (ram_pause_pct / 100)`, the WebUI shows an amber warning banner. The watchdog cannot auto-pause based on RAM% because psutil reports physical RAM only — the banner informs the user they need to PAUSE manually.

**Setup note:** Mac Mini config has `ram_pause_pct: 95`. On a 16 GB machine, threshold = ~15.5 GB. `mistral-nemo:12b` at 128K ctx uses ~25 GB (overflows to swap) → `memory_used_mb > threshold` → banner should appear.

```bash
# Check current threshold and total RAM
curl -s http://localhost:8764/status | python3 -m json.tool | grep -E '"total_mb"|"ram_pause_pct"|"overcommit"'
```

## Test 5 — Feat: CPU/GPU split % in layer ratio line

**What we're testing:** The provider section shows a breakdown of CPU vs GPU memory usage as both MB and percentage of total model size. Uses `size_vram_mb` and `memory_used_mb` from Ollama `/api/ps`.

**Note:** On Apple Silicon (M4) with unified memory, there is no discrete VRAM — `size_vram` will be 0 or nonzero depending on how Ollama allocates GPU layers. The ratio line should only appear when `size_vram_mb` is non-null and non-zero. If `size_vram_mb` is zero, the line is suppressed (CPU-only model — no GPU split to show).

```bash
# Check what Ollama reports for the currently loaded model
curl -s http://localhost:11434/api/ps | python3 -m json.tool | grep -E '"size"|"size_vram"'
```

| 5c | CPU % + GPU % = 100% | ✅ | 57% + 43% = 100% |

## Known issues / deferred

| Issue | Status | Notes |
|-------|--------|-------|
| GPU bar not shown on Apple Silicon (VRAM null) | By design | M4 unified memory not exposed via discrete VRAM API. `size_vram` from Ollama reflects GPU layer allocation within unified memory. |
| CPU/GPU ratio line suppressed when `size_vram = 0` | By design | No ratio to display for CPU-only inference |
| RAM bar underreports load for swap-heavy models on Apple Silicon | Known / documented | psutil reports physical RAM pressure only. Models > physical RAM spill to NVMe swap. Over-commit banner (Test 4) mitigates this. |
| svcmgr/linux.py + windows.py not tested here | Deferred | macOS test target only |
| Integration tests (api.py, ollama.py, watchdog.py) | Deferred | Unit-tested via mocks; real-network integration tests not yet written |
